# Invoicing — Form Discovery

Tool: **invoicing**
Project for testing: **67** (`/67/...`)
Generated by form-discovery agent.

Invoicing spans three sub-domains:
- **Owner invoices** (`owner_invoices` / `payment_applications`) — billed against prime contracts. G702/G703 SOV.
- **Subcontractor invoices** (`subcontractor_invoices`) — billed against commitments (subcontracts / POs). G702/G703 SOV + retainage matrix.
- **Billing periods** + **Payments** (read-only, Acumatica-synced).

The main list lives at `/{projectId}/invoices` with tabs (`?tab=owner|subcontractor|billing-periods|payments`). `/{projectId}/invoicing/*` holds the subcontractor flow. `/{projectId}/billing-periods` is a secondary standalone list.

## Form: owner_invoice_create

- **Title:** New Invoice (owner OR commitment)
- **URL Path:** `/67/invoices/new` (also reachable via `/67/invoices/owner/new` → redirects to `/67/invoices/new?tab=owner`, and `/67/invoicing/new`)
- **File:** `frontend/src/app/(main)/[projectId]/invoices/new/page.tsx`
- **How to Open:** "New Invoice" / "Create Invoice" button on the invoices list (owner tab); or billing-periods row action "Create Invoice" → `/67/invoices/new?tab=owner&billing_period_id=…`.
- **Fields:**

  | Field | Type | Required | Notes |
  |---|---|---|---|
  | Invoice Number | text | yes | |
  | Billing Period | text (free-form, e.g. "January 2026") | yes | parsed to period_start/end client-side |
  | Contract Type | select (Prime / Commitment) | yes | switches contract dropdown + submit path |
  | Contract / Commitment | select | yes | source = `useContracts` (prime) OR `useCommitments` (commitment) |
  | Invoice Date | date popover | yes | |
  | Due Date | date popover | no | |
  | Status | select (draft/submitted/approved/paid/void) | no | mapped to owner/payapp status enums |
  | Description | textarea | no | |
  | Apply retention + Retention % | checkbox + number | no | drives line-item retention calc |
  | Line items (repeating rows) | costCode, description, contractAmount, previouslyBilled, thisMonthAmount + computed | ≥1 | add/remove rows; computed thisMonth%, total, %complete, retention, netDue |

- **Submit Action:** PRIME path: POST `…/contracts/{id}/payment-applications`, then POST `…/invoicing/owner`, then POST per line item `…/invoicing/owner/{id}/line-items`. COMMITMENT path: single POST `/api/invoices`. Then `router.push(/67/invoices)`.
- **Success Criteria:**
  - [ ] New invoice appears in invoices list (owner or subcontractor/commitment tab)
  - [ ] Line items persisted (each row POSTed)
  - [ ] Totals (netDue/retention) match what the summary showed
  - [ ] Error surfaced via InfoAlert + toast on failure (good: `toErrorMessage`)
- **Cleanup:** Delete created owner invoice (+ payment application) / commitment invoice.
- **Risk Notes:**
  - 🚩 **Multi-step non-atomic write (prime path).** Payment-application POST, owner-invoice POST, and N line-item POSTs are sequential with no rollback. A failure mid-loop leaves an orphan invoice with partial line items. Class: line-items-not-fully-persisted.
  - Uses raw `<Table>/<TableBody>/<TableRow>` for the editable SOV grid (allowed here as a form, not a list page).
  - Billing Period is a free-text field parsed heuristically (`resolveBillingPeriodDates`) — "Q1" or odd strings fall back to invoiceDate for both start & end.

## Form: subcontractor_invoice_create

- **Title:** Create New Invoice (subcontractor)
- **URL Path:** `/67/invoicing/subcontractor/new` (optionally `?commitmentId=…&commitmentType=subcontract|purchase_order`)
- **File:** `frontend/src/app/(main)/[projectId]/invoicing/subcontractor/new/page.tsx`
- **How to Open:** Invoices list (subcontractor tab) "Create Invoice"; or from a commitment detail page (passes commitmentId in URL).
- **Fields:**

  | Field | Type | Required | Notes |
  |---|---|---|---|
  | Commitment Type | select (Subcontract / Purchase Order) | yes (if no URL commitment) | picker only shown when no URL commitment |
  | Contract | select | yes | source = `/api/projects/{id}/subcontracts` or `/purchase-orders` |
  | Period Start / End / Billing Date | DateField | no | |
  | Invoice # | text | no | auto-assigned if blank |
  | SOV rows (loaded from commitment) | per-row: work_completed_period $, materials_stored $ | editable cells | scheduled_value/from_previous read-only |
  | Approved CO rows | per-row: work_completed_period $, materials_stored $ (MoneyField) | editable cells | only billed CO rows submitted |

- **Submit Action:** Two buttons — "Save as Draft" (status=draft) and "Submit" (status=under_review). Both POST `…/invoicing/subcontractor/invoices` with `subcontract_id`|`purchase_order_id` + `line_items[]`. Then push to detail page.
- **Success Criteria:**
  - [ ] Invoice created with correct status (draft vs under_review)
  - [ ] SOV line-item edits persisted with correct amounts
  - [ ] Billed CO rows persisted; un-billed CO rows excluded
  - [ ] Redirect to `/67/invoicing/subcontractor/{newId}`
- **Cleanup:** Delete created subcontractor invoice.
- **Risk Notes:**
  - 🚩 **Save handler swallows the real error.** `handleAction` catch does `toast.error("Failed to save invoice")` and discards `err` even though the thrown Error carries the server message. Class: save-handler-swallows-errors.
  - Uses raw `fetch()` (not `apiFetch`) for picklist + commitment loads and submit — bypasses api-client guardrails. ESLint `require-api-client` risk.
  - Raw `<button>` used for the SOV/CO collapse toggles (lint `no-raw-button` risk).
  - Empty-state placeholder uses `SelectItem value="__empty"` — fine, but the contract select is the only FK and sources match the write key, so no FK mismatch.

## Form: owner_invoice_edit (slideover)

- **Title:** Edit Invoice
- **URL Path:** `/67/invoicing/{invoiceId}` → click **Edit** (only when status draft/revise_and_resubmit)
- **File:** `frontend/src/app/(main)/[projectId]/invoicing/[invoiceId]/page.tsx` (`InvoiceEditForm`)
- **How to Open:** Owner invoice detail page, Edit button in header (status-gated).
- **Fields:** invoice_number (text), period_start / period_end (RHFDateField, nullable), status (select), notes (textarea). RHF + Zod (`invoiceEditSchema`).
- **Submit Action:** PATCH `…/invoicing/owner/{invoiceId}`; merges result back into local state; closes slideover; toast success.
- **Success Criteria:**
  - [ ] Reopening Edit shows updated values pre-filled
  - [ ] Status change reflected in header badge + available actions
  - [ ] Toast "Invoice updated successfully"
- **Cleanup:** None (revert by editing back) — or delete the test invoice.
- **Risk Notes:**
  - 🚩 **`notes` defaults to `""` on open, NOT from `invoice.notes`.** Editing without touching notes PATCHes notes=null, silently wiping existing notes. Class: edit-form-pre-fill-bug.
  - 🚩 Catch swallows server error: `toast.error("Failed to update invoice")` discards `err`.

## Cross-cutting risk summary

1. 🚩 **Pervasive error-swallowing catches** — nearly every save handler does `toast.error("Failed to …")` while discarding the thrown Error's server message. Violates CLAUDE.md "Never return generic errors." Best practice already present in `owner_invoice_create` (`toErrorMessage`) and owner email (`reportNonCriticalFailure`).
2. 🚩 **`billing_period_delete` is a no-op** — confirm dialog with no DELETE call.
3. 🚩 **`/billing-periods/new` route missing** — two buttons 404.
4. 🚩 **owner_invoice_edit wipes `notes`** — defaults notes to "" and PATCHes null.
5. 🚩 **owner_invoice_create prime path is non-atomic** — partial-write risk across 3 endpoints.
6. ⚠️ **GeneralTab vs SummaryTab/DetailTab duplication** — confirm GeneralTab is wired or dead.
7. ⚠️ Raw `fetch` (not `apiFetch`) in subcontractor create + SummaryTab; raw `<button>` toggles in subcontractor create.
8. ✅ FK sources match write keys everywhere checked — no dropdown FK-target mismatch in this tool.
