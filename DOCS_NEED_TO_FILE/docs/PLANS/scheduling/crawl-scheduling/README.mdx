# Procore Schedule Management Crawl

This directory contains the comprehensive crawl data for Procore's Schedule Management tool.

## Overview

- **Feature:** Schedule Management (schedulemgmt)
- **Start URL:** https://us02.procore.com/webclients/host/companies/562949953443325/projects/562949954728542/tools/schedulemgmt
- **Project ID:** 562949954728542
- **Company ID:** 562949953443325

## Directory Structure

```
crawl-scheduling/
├── README.md              # This file
├── SCHEDULING-CRAWL-STATUS.md  # Crawl execution status
├── pages/                 # Individual page captures
│   └── {page-name}/
│       ├── screenshot.png
│       ├── dom.html
│       └── metadata.json
└── reports/
    ├── sitemap-table.md   # Overview table of all captured pages
    ├── detailed-report.json  # Full crawl data
    └── link-graph.json    # Page relationship graph
```

## Running the Crawl

```bash
cd scripts/screenshot-capture
node scripts/crawl-scheduling-comprehensive.js
```

## What Gets Captured

1. **Main Schedule Management Page** - Primary entry point
2. **Schedule Views** - Gantt, List, Calendar, Resource views
3. **Task Pages** - Task creation, editing, detail views
4. **Lookahead Views** - Weekly/rolling lookahead schedules
5. **Configuration Pages** - Schedule tool settings
6. **Dropdowns & Menus** - All interactive elements
7. **Tabs** - All tab content variations

## Schedule-Specific Elements Tracked

- Gantt charts
- Timeline views
- Task lists
- Milestones
- Progress indicators
- Dependencies
- Resource assignments
- Date pickers
- Filters and view selectors

## Output Files

- `screenshot.png` - Full page screenshot
- `dom.html` - Raw HTML content
- `metadata.json` - Page analysis including:
  - UI component counts
  - Schedule-specific elements
  - Links and navigation
  - Clickable elements
  - Dropdown menus

## Usage for Implementation

Use the captured data to:
1. Reference UI layouts and component structure
2. Identify required database fields
3. Map navigation patterns
4. Document API endpoints discovered in DOM
5. Create test scenarios based on captured states
