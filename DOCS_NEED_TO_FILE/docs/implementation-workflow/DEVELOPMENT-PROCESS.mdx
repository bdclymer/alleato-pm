# Alleato-Procore Development Process

## The Problem We're Solving

We have 570MB of crawled Procore data (screenshots, HTML, metadata) across 22 features, but:
- It's overwhelming and hard to navigate
- No clear process for turning crawled data into implementation
- No visibility into how UI connects to database during development
- Hard to verify if our schemas actually match Procore's functionality

This document defines **THE STANDARD PROCESS** for implementing any Procore feature.

---

## Development Philosophy

```
Procore Screenshots/HTML → Understand UI → Derive Schema → Implement → Verify Match
         ↓                                      ↓
  Support Docs (RAG)                   Development Annotations
```

**Key Insight:** The UI drives the schema, not the other way around. We study what Procore shows, then figure out what database structure supports it.

---

## The 6-Phase Feature Development Process

### Phase 1: Discovery (Before Writing Any Code)

**Goal:** Understand exactly what Procore does for this feature.

#### Step 1.1: Locate Feature Documentation
```
apps/docs/pages/PLANS/{feature-name}/
├── README.md                    # Start here - feature overview
├── PLANS.md                     # Implementation roadmap
├── {FEATURE}-CRAWL-STATUS.md    # Technical analysis of crawl
├── COMPARISON-REPORT.md         # Procore vs current implementation gaps
├── procore-{feature}-crawl/     # Screenshots and HTML
│   ├── pages/                   # Individual page captures
│   └── reports/                 # Analysis reports
└── forms/                       # Form field captures (CSV)
```

#### Step 1.2: Review Visual References
```bash
# Open the feature's screenshot directory
open "apps/docs/pages/PLANS/{feature}/procore-{feature}-crawl/pages"

# Each page folder contains:
# - screenshot.png (what user sees)
# - dom.html (HTML structure)
# - metadata.json (extracted components: buttons, forms, tables)
```

#### Step 1.3: Query Support Documentation (RAG)
Use the Supabase vector store with embedded Procore support docs:
- Search for business rules and workflows
- Clarify field meanings and relationships
- Understand edge cases

#### Step 1.4: Create Feature Spec
Output: `{feature}/SPEC.md` containing:
- List of all UI screens/states
- All form fields with types and validation rules
- All table columns with data sources
- Calculated fields and their formulas
- Status workflows
- Relationships to other features

---

### Phase 2: Schema Design

**Goal:** Design database schema that supports the UI requirements.

#### Step 2.1: Analyze Form Fields → Table Columns
From crawled forms, extract:
```markdown
| UI Field | DB Column | Type | Required | Notes |
|----------|-----------|------|----------|-------|
| Contract # | number | text | Yes | Auto-generated |
| Title | title | text | Yes | Max 255 chars |
| Original Value | original_contract_value | numeric | Yes | Currency |
| Revised Value | revised_contract_value | numeric | CALCULATED | = original + approved_cos |
```

#### Step 2.2: Identify Calculated Fields
Procore shows many calculated/aggregated values. These should be:
- **Database VIEWs** for complex aggregations
- **Generated columns** for simple calculations
- **NEVER** stored redundantly (violates normalization)

#### Step 2.3: Map Relationships
```
prime_contracts
  └── contract_line_items (SOV)
       └── owner_invoice_line_items
  └── contract_change_orders
  └── owner_invoices
       └── payment_applications
```

#### Step 2.4: Create Migration
Output: `supabase/migrations/{timestamp}_{feature}_schema.sql`

### Phase 3: Implementation with Dev Annotations

**Goal:** Build UI with visible connections to data sources.

#### The Dev Annotation System

During development, add data-source annotations to components that show:
- Which database table/view feeds this component
- Which columns are being displayed
- Relationship paths for related data

```tsx
// Example: Prime Contracts list with dev annotations
<DataTable
  data={contracts}
  // DEV: Remove before production
  data-dev-source="prime_contracts_with_financials (view)"
  data-dev-query="SELECT * FROM prime_contracts_with_financials WHERE project_id = ?"
>
  <Column
    header="Contract #"
    accessor="number"
    data-dev-column="prime_contracts.number"
  />
  <Column
    header="Revised Value"
    accessor="revised_contract_value"
    data-dev-column="CALCULATED: original_contract_value + SUM(change_orders.approved_amount)"
  />
  <Column
    header="% Paid"
    accessor="percent_paid"
    data-dev-column="CALCULATED: total_paid / revised_contract_value * 100"
  />
</DataTable>
```

#### Dev Mode Toggle
Add a development mode that:
1. Shows data source badges on components
2. Displays the actual SQL query being executed
3. Shows which table/column each field maps to
4. Highlights calculated vs stored fields

```tsx
// In your app, add a dev mode context
const DevModeContext = createContext({ enabled: false });

// Component that shows source info in dev mode
function DevSourceBadge({ table, column, calculation }) {
  const { enabled } = useContext(DevModeContext);
  if (!enabled) return null;

  return (
    <div className="absolute -top-6 left-0 text-xs bg-yellow-100 px-2 py-1 rounded">
      {calculation ? `CALC: ${calculation}` : `${table}.${column}`}
    </div>
  );
}
```

#### Why This Matters
- **You** can see exactly where data comes from while developing
- **New developers** can understand the data flow immediately
- **Schema accuracy** becomes visible - if UI shows "Revised Value" but you're not calculating it, the badge makes that obvious
- **Remove before production** with a single flag flip

### Phase 4: Verification

**Goal:** Prove implementation matches Procore.

#### Step 4.1: Side-by-Side Comparison
```bash
# Generate comparison screenshots
npm run test:visual -- --feature={feature}

# Output:
# - procore-screenshot.png (from crawl)
# - implementation-screenshot.png (from test)
# - diff.png (visual diff)
```

#### Step 4.2: Field-by-Field Audit
Update `COMPARISON-REPORT.md` with:
- Each Procore field → our implementation status
- Score: % of Procore functionality matched
- Gaps identified with severity

#### Step 4.3: Calculate Match Score
```markdown
| Category | Weight | Score | Grade |
|----------|--------|-------|-------|
| List Table Columns | 35% | 50% | F |
| Form Fields | 25% | 55% | F |
| Financial Calculations | 25% | 0% | F |
| Database Schema | 10% | 60% | D- |
| Status Workflow | 5% | 70% | C- |
| **OVERALL** | **100%** | **40.8%** | **F** |
```

---

### Phase 5: Testing

**Goal:** Automated verification that implementation works.

Use playwright-tester sub-agent:
```typescript
Task({
  subagent_type: "test-automator",
  prompt: `Test {feature} implementation against Procore reference.

  REFERENCE: apps/docs/pages/PLANS/{feature}/procore-{feature}-crawl/

  TESTS:
  1. All form fields accept valid input
  2. All table columns display correct data
  3. Calculated fields compute correctly
  4. CRUD operations work
  5. Status workflow transitions work

  OUTPUT: HTML test report with screenshots`
})
```

## Current State: IMPLEMENTING / COMPLETE / BLOCKED

## For New Team Members

### Day 1 Orientation
1. Read this document
2. Pick a feature from `apps/docs/pages/PLANS/`
3. Open its `README.md`
4. Look at screenshots in `procore-{feature}-crawl/pages/`
5. Read `COMPARISON-REPORT.md` to see gaps
6. Check `STATUS.md` for current state

### Before Starting Any Feature
1. **READ** the feature's documentation folder completely
2. **VIEW** all crawled screenshots
3. **UNDERSTAND** what Procore does before writing code
4. **CHECK** the comparison report for known gaps
5. **ASK** if requirements are unclear

### Definition of Done
- [ ] Match score ≥ 80% (Grade B or better)
- [ ] All critical fields implemented
- [ ] Calculated fields compute correctly
- [ ] E2E tests pass
- [ ] COMPARISON-REPORT.md updated
- [ ] STATUS.md shows "COMPLETE"

## Summary

1. **Discovery** → Understand Procore from crawled data
2. **Schema Design** → Derive database from UI requirements
3. **Implementation** → Build with dev annotations showing data sources
4. **Verification** → Compare against Procore screenshots
5. **Testing** → Automated E2E tests
6. **Documentation** → Update status and match score

**The dev annotation system** is the key insight - it gives you visibility during development without polluting production code.
