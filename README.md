# Alleato-PM

Alleato-Procore is the internal repo that powers the Alleato Project Manager add-on for Procore. It bundles a Next.js 15 frontend, a Python backend, Supabase schema management, a suite of Playwright-driven automation, and rich documentation + PRP planning content so every contributor stays grounded in Procore’s workflows.

## Quick start

### Prerequisites

- Node.js 18+ (LTS) and npm (pnpm is supported inside `frontend/` as well)
- Python 3.11+ for the backend services (see `backend/requirements.txt`)
- Access to the Supabase workspace described by `frontend/.env.example`
- OpenAI API key (used by multiple AI-enhanced helpers)

### One-time setup

```bash
git clone <repo-url>
cd alleato-procore
npm run install:all   # installs root + frontend deps
cp frontend/.env.example frontend/.env.local
cp backend/.env.template backend/.env
```

Edit the copied `.env*` files with Supabase credentials, OpenAI keys, and any overrides described inside `frontend/.env`, `frontend/.env.local`, and `backend/.env`. The frontend config also reads `.env.local` inside `frontend/` and the root `.env` when running Playwright from `frontend/config/playwright/`.

### Running locally

- Start backend alone: `npm run dev:backend`
- Start frontend dev server: `npm run dev:frontend`
- Start both services together (concurrently): `npm run dev`
- Generate schema docs after DB changes: `npm run schema:docs`
- Seed Supabase data: `npm run seed:db`

## Repository tour

- `frontend/` – Next.js 15 App Router application (components scoped by domain) plus all UI assets, Playwright tests (`frontend/tests`, `frontend/tests/e2e`), multiple `frontend/config/playwright/*.ts` variations, and additional tooling.
- `backend/` – Python FastAPI services, Supabase-connected logic, and pytest suites.
- `supabase/` – SQL migrations, Flake8/RLS policies, and helpers that keep the database schema in sync.
- `scripts/` – Automation (Playwright crawls, schema extractors, seeders, quality gates, supporting utilities).
- `tests/` – Repository-wide suites such as `tests/e2e` (uses the root `playwright.config.ts`) and `tests/supabase-e2e` for integration checks.
- `docs/` – Living technical documentation (start with `docs/development/DEVELOPER_MANUAL.mdx` and `docs/index/PROCORE-TUTORIALS.md`).
- `docs-ai/` – Autogenerated documentation content and curated PRP outputs produced via Claude/AI workflows.
- `PRPs/` – Feature research, crawls, spec output, and verification reports grouped by domain (finance, PM, core tools).
- `screenshots/` & `scripts/screenshot-capture/` – Documentation artifacts and visual regression captures referenced in PRPs.
- `.claude/` – Workflow automation, gate enforcement, templates, and agent tooling (do not delete).
- `playwright/`, `frontend/config/playwright/`, and `frontend/playwright.config.ts` – Playwright config + rerouted entries for the frontend-specific suites.
- `/redoc` – ReDoc-powered single-page reference covering authentication schemes, schema definitions, and every documented frontend/backend API path.
- `/api-docs` – Client-side Swagger UI powered by `/openapi.json`/`/openapi.yaml`, so you can explore both the FastAPI backend and Next.js frontend REST surfaces directly in the app.

## Testing & verification

- Frontend unit: `npm run test:frontend` (runs the frontend package’s unit tests)
- Backend: `npm run test:backend`
- Full suite (includes Playwright): `npm run test` or `npm run test:all` (`scripts/run-all-tests.sh`)
- Playwright UI tests (frontend): `cd frontend && npx playwright test tests/e2e/*.spec.ts --config=config/playwright/playwright.config.ts`
- Root-level Playwright (`tests/e2e`): `npx playwright test tests/e2e` (uses `playwright.config.ts` at the repo root)

## Collaboration pointers

1. **Branch strategy** – Create feature branches like `claude/<feature>-<ticket>` off `origin/main` (or work from a fork if you lack push access). Keep your branch rebased before opening a PR.
2. **Supabase gate** – Follow `.claude/rules/SUPABASE-GATE.md` and the scaffolds under `.claude/scaffolds/` when you modify schema-related files; the husky hook (`scripts/enforce-gates.ts`) blocks commits until the gate is satisfied.
3. **Focus on PRPs** – Document your feature in `PRPs/<category>/<feature>/prp-*.md`, link to the corresponding crawl folder, and capture any automation requests in `.claude/templates/tasks`.
4. **Testing & artifacts** – Run the relevant unit/Playwright suites after changes. Collect artifacts (`frontend/playwright-report/`, `screenshots/`, `tests/test-results-direct-costs.json/`) and attach them to PRPs when required.
