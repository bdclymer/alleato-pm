## CRITICAL FINDINGS â€” needs human review

### 1. `subcontract_sov_items.budget_code` â€” text column, but BudgetCodeSelector returns `project_budget_codes.id`
**File:** `frontend/src/components/domain/contracts/subcontract-form/SovTableRows.tsx`
- DB column `subcontract_sov_items.budget_code` is plain `string | null` with **no FK constraint**.
- Form passes `value={line.budgetCodeId || line.budgetCode || ""}` to `BudgetCodeSelector`.
- BudgetCodeSelector match key is `id` (i.e. `project_budget_codes.id`), but the DB stores the budget code TEXT (e.g. `"01-100"`).
- A reconciliation effect lives in `useSubcontractFormState.ts:242-258` that retrofits `budgetCodeId` from `budgetCode` text on edit by matching `c.code === storedCode`.
- **Status: RESOLVED via reconciliation effect**, but the pattern is fragile: if a budget code text exists in the DB row that no longer matches any current `project_budget_codes.code`, the dropdown will show empty on edit.

**Recommendation:** keep as-is, but add an integration test that exercises edit for a subcontract whose stored `budget_code` text is missing from `project_budget_codes`.

### 2. `commitment_change_orders` (commitment CO new line items) â€” RESOLVED but writes through commitment-options view IDs
**File:** `frontend/src/app/(main)/[projectId]/change-orders/commitment/new/page.tsx`
- `CommitmentCombobox` reads from `/api/projects/[id]/commitment-options` which queries `commitments_unified` view.
- `commitments_unified.id` returns either `subcontracts.id` or `purchase_orders.id` directly.
- The form writes `contract_change_orders.contract_id`, which has **NO declared FK relationship** in the schema.
- **Status: MATCH** â€” both write and read use the unified view, so IDs are consistent.

### 3. `change_event_line_items` (Budget Code â†’ budget_lines vs project_budget_codes / Vendor â†’ companies vs vendors)
**File:** `frontend/src/components/domain/change-events/change-event-form/LineItemRow.tsx`
- Known historical mismatch documented in `docs/patterns/form-id-mismatch-prevention.md`.
- Verified RESOLVED in `frontend/src/app/api/projects/[projectId]/change-events/[changeEventId]/line-items/route.ts`:
  - Read path: `budget_lines.id` is mapped to `project_budget_codes.id` via `budget_lines.project_budget_code_id`.
  - Write path: `vendor_id` stores `companies.id` directly (vendor combobox uses `/api/projects/.../vendors` which already returns `companies.id`).
- **Status: RESOLVED.**

## FRAGILE PATTERN â€” keep an eye on

### 5. `members-tab.tsx` matches permission template by `name`, not by ID
**File:** `frontend/src/app/(main)/[projectId]/admin/_components/members-tab.tsx:202`
- `value={templates.find((t) => t.name === member.template_name)?.id ?? ""}`
- Reads `member.template_name` (string) and finds the matching template by name to get the ID for the dropdown.
- Writes the template ID via `onAssignTemplate`.
- **Risk:** if two templates ever share the same name, or if a template is renamed but the old name persists on a member record, the dropdown will silently show empty/wrong selection.
- **Recommendation:** modify the API to return `permission_template_id` (UUID) on the member object directly. Use ID-to-ID matching, not name-to-ID.

## Top fixes by impact

Ranked by number of forms each issue affects and severity:

1. **(orphan)** Delete or refactor `ChangeEventLineItemsGrid.tsx` â€” it duplicates `LineItemRow.tsx` without the resolution mapping that protects against the budget_code/vendor mismatch. If wired in, the bug recurs. **Single file, dormant risk, easy fix.**

2. **(fragile)** Fix `members-tab.tsx` template matching by ID instead of name â€” return `permission_template_id` from the project members API. **Single file, low priority, but fragile.**

3. **(test gap)** Add an integration / Playwright test that loads an edit form for: subcontract with a stored `budget_code` text that no longer maps to `project_budget_codes` â†’ the row currently re-renders empty. The reconciliation effect in `useSubcontractFormState.ts` fixes the common case but fails for stale text. **Affects subcontract edit + purchase-order edit (both use text-based FK-less storage).**

4. **(documentation)** Update `docs/patterns/form-id-mismatch-prevention.md` to note that:
   - The historical `vendor â†’ companies vs vendors` mismatch is now resolved at the API level: `/api/projects/[id]/vendors` returns `companies.id` (no separate `vendors` table exists).
   - The `subcontract_sov_items.budget_code` and `purchase_order_sov_items.budget_code` columns are TEXT (not FK), so the form intentionally round-trips by code-string, and the reconciliation effect is the canonical pattern for these cases.

5. **(architecture risk)** The schema has multiple UUID columns intended as FKs but with NO declared FK constraint:
   - `subcontracts.contract_company_id` (writes companies.id by convention)
   - `contract_change_orders.contract_id` (writes subcontracts.id or purchase_orders.id by convention)
   - `punch_items.{assignee_id, ball_in_court_id, final_approver_id, punch_item_manager_id}` (writes auth_user_id by convention)
   - `change_event_rfqs.assigned_contact_id` (writes people.id by convention)
   These are not bugs today, but they leave the door open for orphaned rows. **Recommend adding FK constraints in a future migration.**

## Methodology notes

- Read `frontend/src/types/database.types.ts` once and used grep for FK lookups (file is 30k lines).
- For each form file: identified `<Select>`, `<Combobox>`, `<BudgetCodeSelector>`, `<VendorCombobox>`, `<ContactCombobox>`, `<CommitmentCombobox>`, `SearchableSelect`, and `RHFComboboxField` usages that bind to a column whose value is a UUID.
- For each, traced the Source API or hook (e.g. `useCompanies`, `useAuthUsers`, `useContracts`, `/api/projects/[id]/vendors`) to determine the source table.
- Cross-referenced the storing table's `Relationships` array in `database.types.ts` to find the FK target.
- For declared FK mismatches, searched the corresponding API route for resolution mapping (`map`, `resolve`, `reverse`, references to both tables) before classifying as RESOLVED vs NEEDS FIX.

Audit performed by Claude (read-only, no source code modified, no DB types regenerated).
