# Code-Mode RPC Security Decision

Date: 2026-06-19
Owner: Codex / Alleato AI
Related Linear: AAI-566
Related goal: `docs/ai-plan/hermes-openclaw-goals/goal-07-later-high-risk-work.md` C10

Decision: Not approved for implementation.

## Decision Summary

Do not build Code-Mode RPC yet. Hermes proves the pattern can reduce token use
by letting a generated script call tools through RPC and return compact stdout,
but the same pattern creates an arbitrary-code-execution boundary inside the AI
assistant. Alleato should not expose that boundary until the sandbox is isolated
from production credentials, the tool RPC surface is read-only by default, and
escape tests are automated.

The approved next step is a security spike only. The implementation can be
reconsidered after the controls below are satisfied and this decision is changed
to `Approved for implementation`.

## Source Usage

Hermes source usage: REFERENCE only.

- `hermes-agent/tools/code_execution_tool.py` was reviewed for the architecture:
  generated tool stubs, parent-process RPC dispatch, environment scrubbing,
  strict tool allowlist, max tool calls, timeouts, stdout/stderr truncation,
  secret redaction, and local/remote sandbox transports.
- No Hermes Python runtime, transport, subprocess wrapper, terminal backend,
  file-based RPC loop, UDS listener, or tool stub code is copied into Alleato.

## Threat Model

The untrusted inputs are model-generated scripts, tool arguments generated by
those scripts, and any data returned by tools into the script. The protected
assets are Supabase service-role credentials, Microsoft/Teams/Outlook tokens,
Render environment variables, local filesystem contents, private network
endpoints, production database mutation paths, delivery tools, and user-visible
assistant output.

Primary risks:

- Secret exfiltration through environment variables, stdout, stderr, tool input,
  tool output, trace metadata, or network calls.
- Filesystem escape through path traversal, symlink following, repo writes, or
  generated patch/write tools.
- SSRF or metadata-service access from script-controlled URLs.
- Privilege escalation by calling write, delivery, terminal, patch, or MCP tools
  outside the normal assistant policy layer.
- Denial of service through infinite loops, unbounded output, recursive tool
  calls, high-cost retrieval loops, or long-running scripts.
- Silent partial execution where the assistant reports success after the script
  timed out, hit a policy denial, or received truncated output.

## Required Sandbox Decision Before Build

A future implementation must choose one of these explicitly:

1. External isolated worker: preferred. Run code in a short-lived Render job or
   equivalent locked-down container with no production secrets except an
   ephemeral RPC token.
2. No implementation: acceptable if the workflow can be solved by normal tool
   planning plus context compaction.

In-process `eval`, `Function`, `vm`, local shell execution, Next.js route
subprocesses, and same-process RPC listeners are rejected for production.

## Minimum Controls For Approval

- Feature flag: default-off `AI_ASSISTANT_CODE_MODE_RPC_ENABLED`.
- Runtime isolation: no execution in the Next.js server process; no inherited
  production environment; no mounted repo write access.
- Environment scrubbing: exact allowlist only. Deny names containing `KEY`,
  `TOKEN`, `SECRET`, `PASSWORD`, `AUTH`, `DSN`, `COOKIE`, `WEBHOOK`, `CLIENT`,
  `SUPABASE`, `OPENAI`, `AI_GATEWAY`, `MICROSOFT`, `TEAMS`, `OUTLOOK`, or
  `VERCEL` unless the variable is an explicit ephemeral sandbox value.
- Tool allowlist: read-only first release. No `terminal`, `write_file`, `patch`,
  delivery, email-send, Teams-send, mutation, Supabase admin, MCP mutation, or
  browser-control tools.
- Registry integration: callable tools must come from the existing assistant
  tool registry and pass workflow, actor mode, source-family, project-scope,
  write-permission, and delivery-permission filters.
- Network policy: any URL-bearing tool must continue through the OpenClaw
  egress guard. Metadata, loopback, private, link-local, and special-use IPs
  remain blocked.
- Output policy: stdout and stderr are capped, stripped of ANSI/control
  sequences, redacted with the existing secret redaction policy, and marked as
  truncated when applicable.
- Limits: max wall-clock timeout, max tool calls, max per-tool timeout, max
  stdout/stderr bytes, max request/response bytes, and max script source bytes.
- Observability: every run records status, reason, tool-call count, duration,
  truncation flags, denial reason, and sanitized output in the AI Ops ledger.
- Failure-loud behavior: disabled, unsupported tool, policy denial, timeout,
  sandbox startup failure, tool-call-limit exhaustion, output truncation, and
  redaction failure must return typed statuses. None can be reported as success.

## Required Escape Tests Before Approval

- Environment secret probe: script tries to print env vars matching secret names;
  output must be redacted or unavailable.
- Filesystem probe: script tries to read `.env`, `.env.local`, `.npmrc`,
  Supabase temp files, and parent directories; all must be denied.
- Network probe: script or URL tool tries `169.254.169.254`, loopback, private
  IPs, IPv6 mapped loopback, and obfuscated IPv4 literals; all must be denied.
- Tool allowlist probe: script tries write, delivery, terminal, patch, browser,
  Supabase admin, and MCP mutation tools; all must be denied with reason codes.
- Timeout probe: infinite loop and slow tool call both terminate with timeout
  status and ledger evidence.
- Output cap probe: oversized stdout/stderr is truncated, redacted, and marked.
- Recursive RPC probe: script attempts to call Code-Mode RPC from inside itself;
  nested execution is denied.
- Persistence probe: every failed and denied run records an AI Ops ledger row.

## Current Approval Status

Current status: deferred.

Rationale: the repo has strong prerequisites from Goals 1-6, including egress
guardrails, outbound action policy, tool registry filtering, and context
compaction. It does not yet have an approved isolated runtime, an explicit
read-only Code-Mode tool allowlist, or automated sandbox escape tests.

## Guardrail

`scripts/verify/verify_code_mode_rpc_guardrail.mjs` fails while this decision is
not approved if Code-Mode RPC runtime paths or the default feature flag appear
under app/backend source. This keeps the required security decision from being
bypassed by implementation drift.
