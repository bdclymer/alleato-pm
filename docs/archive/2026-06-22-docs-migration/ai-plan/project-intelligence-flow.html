Project Intelligence

What the compiler is compiling, what gets stored, and what the assistant loads later

The compiler is not compiling code. It is compiling raw project signals into durable, project-level intelligence. The durable output is a set of insight_cards plus one current intelligence_packet per project target. Then, when the assistant is asked about a project, it loads that packet first and layers live tools on top.

Built on demand

What happens here

    Clean Flow

    Left side is created ahead of time by ingestion and compiler jobs. Right side is assembled when the assistant answers a project question.

    Ahead Of Time
    1. Source lands

    document_metadata and other raw sources

    Fireflies meetings, emails, Teams, files, extracted tasks, and structured project facts arrive first.

    meeting transcript email Teams document task
    2. Project target exists

    intelligence_targets

    One target row represents the thing we are compiling intelligence for. For project briefings, this is the client_project target tied to a project_id.

    3. Weak extracted signal is staged

    source_signal_candidates

    The compiler writes candidate signals first. This is the “maybe this matters” layer before durable project intelligence is created.

    4. Candidate is promoted

    insight_cards + insight_card_targets + insight_card_evidence

    Promotion turns a candidate into a durable insight, links it to the project target, and stores the exact supporting evidence rows.

    • insight_cards: the durable conclusion
    • insight_card_targets: which project it belongs to
    • insight_card_evidence: which meeting/email/task/doc supports it
    5. Refresh is queued

    packet_refresh_jobs

    Every successful promotion queues a project packet refresh so the assistant has a current summary packet to load later.

    6. Current packet is rebuilt

    intelligence_packets + intelligence_packet_cards

    The packet compiler assembles the selected project’s active cards into a current reusable packet with executive summary, recommended next moves, source coverage, and ordered packet sections.

    At Answer Time
    A. User asks about a project

    Resolve selected project

    The assistant determines which project the user means, then resolves that project’s intelligence_target.

    B. Preload project intelligence

    loadIntelligencePacket(projectId)

    The assistant loads the current packet first. This is the “expert starting context” layer you were asking for.

    C. Pull live project facts

    getProjectBriefingSnapshot(projectId)

    This is a separate live snapshot tool. It pulls budget, commitments, change orders, RFIs, submittals, schedule, notifications, and recent docs directly from operational tables.

    D. Fill in what the packet does not cover

    RAG and source-specific search

    If the user asks for exact proof, exact spec text, the latest meeting detail, or something stale/missing, the assistant still calls semantic search and source-specific tools instead of trusting the packet alone.

    E. Build live timeline and warnings

    Timeline + stale/missing warnings

    Recent timeline entries and warning banners are assembled on demand. This is where the assistant can say “packet loaded,” “packet missing,” “snapshot missing,” or “document intelligence is thin.”

    F. Final answer

    Packet first, live verification second

    Best-case answer path: start from the packet for broad context, then confirm specifics with live tools before stating exact dates, numbers, obligations, or quotes.

    What The End Result Is Supposed To Be

    For a selected project, the assistant should already have a compact expert briefing packet before it starts searching.

    Start with current packet Know the latest major shifts Know the open risks and next moves Know what evidence supports each claim Call RAG only when deeper proof is needed
    In other words: the assistant should not begin from zero. It should begin with a durable project briefing, then verify and deepen from live sources.

    Best Operating Model

    This is the simplest durable behavior if you want the assistant to start with expert project context.

    Recommended

    Keep one durable current packet per project

    This is effectively the machine-readable version of the running project intelligence document you described.

    Recommended

    Load that packet automatically when the assistant is clearly project-scoped

    That gives the assistant a strong starting context instead of forcing it to rediscover the whole project every time.

    Required

    Do not trust the packet alone for “today,” exact numbers, or exact proof

    The packet is a briefing layer. Live source tools and RAG still need to verify the latest meeting details, current obligations, and exact citations.

    Required

    Freshness has to be truthful

    Right now the packet freshness flag can say “fresh” even when a newer meeting has landed and has not been packetized yet. That needs to be fixed if you want reliable high-quality answers.
